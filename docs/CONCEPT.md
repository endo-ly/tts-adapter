## このRepoについて

`tts-adapter` は、TTSエンジンを直接使うためのライブラリではなく、**複数のTTSエンジンをエージェントや外部ツールから同じ形で扱うためのアダプタサーバー**である。

TTSエンジンは、それぞれAPI形式、モデル指定、声の指定方法、実行方法が異なる。あるエンジンでは `voice` を文字列で指定し、別のエンジンでは参照音声やlatentファイルを指定し、さらに別のエンジンではcaptionやspeaker idを使う、といった違いがある。

この差分をクライアント側に直接持ち込むと、OpenClaw、自作エージェント、UI、通知処理などが、それぞれTTSエンジン固有の事情を知る必要がある。

`tts-adapter` はその差分を吸収し、クライアント側からは次のように扱えるようにする。

```json
{
  "model": "tts-default",
  "voice": "your-voice-name",
  "input": "こんにちは"
}
```

つまり、このRepoの目的は **「TTSを動かすこと」そのものよりも、「声の出口を安定したAPIとして固定すること」** である。

## なぜ必要か

AIエージェントに音声を持たせる場合、単にTTSを1回呼べればよいわけではない。

次のような問題が出てくる。

* OpenClawなど、OpenAI互換APIしか想定していないクライアントからローカルTTSを使いたい
* 自作エージェントからは、OpenAI互換にない独自パラメータも渡したい
* 特定の人格声を、TTSエンジンに依存しないIDとして管理したい
* Irodoriから別のTTSに変えても、クライアント側の設定をなるべく変えたくない
* TTSのCLI実行、timeout、tmpファイル管理、同時実行制御などをクライアント側に書きたくない
* 複数のエージェントやUIから、同じ音声生成口を使いたい

`tts-adapter` は、これらをまとめて受けるための薄い常駐APIサーバーである。

## どんな時に使うか

### OpenAI互換TTSとしてローカルTTSを使いたい場合

OpenClawなど、OpenAI互換のTTS APIを前提にしているツールから、IrodoriなどのローカルTTSを利用したい場合に使う。クライアント側は `baseUrl` を `tts-adapter` に向けるだけで、OpenAI互換TTSとして扱える。

### 自作エージェントに安定した音声出力口を持たせたい場合

自作エージェントから直接Irodoriや各TTSエンジンを呼ぶのではなく、`/v1/speech` を通して音声生成を行う。エージェント本体はTTSエンジン固有の実行方法を知らなくてよい。

### 人格声を固定IDで管理したい場合

`voice_id` は、TTSエンジン固有の声指定ではなく、論理的な声IDとして扱う。たとえば `your-voice-name` というvoiceを、現在は [Irodori-TTS](https://github.com/Aratako/Irodori-TTS) の `ref_latent.pt` で実現し、将来は別のTTS Providerで実現する、といった運用が可能。

### TTSエンジンを将来差し替えたい場合

最初はIrodoriを使い、将来Qwen-TTSや別のTTSエンジンを追加したい場合に、Providerとして拡張できる。クライアント側は `model = tts-default` や `voice = your-voice-name` を使い続け、内部のProviderだけを切り替えることを目指す。

## 使わないほうがよいケース

このRepoは、すべてのTTS用途に向いているわけではない。

次のような場合は、TTSエンジンを直接使ったほうが単純である。

* 1つのTTSエンジンだけを、単発のスクリプトから呼べればよい
* OpenAI互換APIが不要
* voice / model / provider を分離して管理する必要がない
* エージェントや複数クライアントから共通利用する予定がない
* CLIで手動生成できれば十分

`tts-adapter` は、単発の音声生成ツールではなく、**エージェントシステム向けの音声出力レイヤー**として使うことを想定している。
